# Отчёт: база данных университета с шардингом

## 1. Дана схема базы данных `university_grades` с прошлого задания

- **Назначение**: хранение данных о студентах, курсах, преподавателях, оценках и записях на курсы.
- **Коллекции**:
  - **students**: сведения о студентах (`student_id`, ФИО, контакты, факультет, группа, год поступления и т.п.).
  - **courses**: курсы (`course_code`, название, факультет, кредиты и т.п.).
  - **lecturers**: преподаватели (`lecturer_id`, ФИО, кафедра и т.п.).
  - **grades**: оценки по курсам (`student_id`, `course_code`, `grade`, тип оценки, семестр, учебный год, дата).
  - **enrollments**: записи на курсы (`student_id`, `course_code`, учебный год, семестр, статус и т.п.).

**Индексы:**

- `students.student_id` — уникальный.
- `courses.course_code` — уникальный.
- `lecturers.lecturer_id` — уникальный.
- `grades.(student_id, course_code)` — для выборок оценок по студенту и курсу.
- `enrollments.(student_id, course_code)` — для выборок записей.

Шард-ключ для коллекций `students`, `grades`, `enrollments` выбран по полю **`student_id` (hashed)**, так как типичные запросы строятся вокруг студента (оценки, записи на курсы).


## 2. Реализация шардинга

### 2.1. Архитектура кластера

Развёртывание через `docker-compose`:

- **Config Server**: сервис `configsvr`, реплика-сет `cfgRS`, порт 27100.
- **Шарды**: `shard1` (shard1RS, порт 27101), `shard2` (shard2RS, порт 27102); по одному узлу в каждом реплика-сете.
- **Маршрутизатор**: `mongos-router`, порт 27018, `--configdb cfgRS/mongo_configsvr:27017`.
- **Инициализация**: сервис `mongo-setup` выполняет скрипты в `scripts/` (init-configrs.js, init-shard1rs.js, init-shard2rs.js, init-router.js) — инициализация реплика-сетов, добавление шардов в mongos, включение шардинга для БД `university_grades`, шардирование коллекций по `student_id` (hashed). Коллекции `courses` и `lecturers` не шардируются.

### 2.2. Обоснование выбора шард-ключа

- Запросы в основном по студенту; hashed-шардирование по `student_id` даёт равномерное распределение и targeted-запросы на один шард при `find({ student_id: "..." })`.
- Справочники `courses` и `lecturers` малы по объёму и остаются на одном шарде.

## 3. Нагрузочное тестирование

### 3.1. Цель

Сравнить производительность **до шардинга** (одиночный инстанс MongoDB) и **после шардинга** (кластер с mongos и двумя шардами) на одном и том же сценарии нагрузки, записать результаты в DataFrame и визуализировать сравнение на графиках matplotlib.

### 3.2. Порядок тестирования
   - Сначала выполняется нагрузочный тест против **одиночного инстанса** (URI с портом 27019 / сервис `mongodb_single`): многопоточное чтение оценок по случайному `student_id` из коллекции `grades`. Результаты (latency_ms, success, op_index) записываются в DataFrame с меткой кластера `"single"`.
   - Затем тот же сценарий выполняется против **шардированного кластера** (URI с портом 27018 / сервис `mongos_router`). Результаты записываются в DataFrame с меткой `"sharded"`.
   - Оба DataFrame объединяются; по ним считаются пропускная способность (ops/sec) и статистики задержек (средняя, медиана, 95-й перцентиль) для каждого варианта.
   - Строятся два графика matplotlib:
     - **Bar-диаграмма**: распределение задержек (latency_ms) по кластерам (single vs sharded).
     - **Bar-диаграмма**: сравнение пропускной способности (ops/sec) для single и sharded.

### 3.3. Параметры теста
   - `--threads` — число потоков (по умолчанию 8).
   - `--ops-per-thread` — число операций на поток (по умолчанию 1000). Общее число операций = threads × ops_per_thread для каждого кластера.

### 3.4. Сценарий нагрузки
   - Сбор до 1000 значений `student_id` из коллекции `students`.
   - В каждом потоке: многократный вызов `grades.find({ student_id: <случайный id> })` и замер времени отклика. Имитируется типичный запрос «все оценки студента».

### 4. Результаты тестирования

Параметры прогона: 8 потоков, 1000 операций на поток (всего 8000 операций на каждый кластер).

| Метрика                      | Одиночный инстанс (single) | Шардированный кластер (sharded) |
|-----------------------------|----------------------------|----------------------------------|
| Всего операций              | 8000                       | 8000                             |
| Общее время, сек            | 1.984                      | 2.236                            |
| Пропускная способность, ops/sec | 4031.53                | 3577.22                          |
| Средняя задержка, мс        | 1.97                       | 2.21                             |
| Медиана задержки, мс        | 1.79                       | 2.08                             |
| 95-й перцентиль, мс         | 3.60                       | 3.61                             |

**Примеры из DataFrame (первые строки):**

- single: задержки в диапазоне примерно 1–5 мс (например, 3,86; 3,87; 1,28; 1,05; 5,07 мс).
- sharded: задержки выше и с большим разбросом (например, 8,93; 11,88; 3,78; 2,08; 10,51 мс), что отражает дополнительный hop через mongos и маршрутизацию по шардам.

**Графики:**

- График 1: столбчатая диаграмма средней задержки (мс) — сравнение single vs sharded.
- График 2: столбчатая диаграмма пропускной способности (ops/sec) — сравнение single vs sharded.

**Выводы:**

В проведённом тесте при развёртывании на одной машине и умеренном объёме данных **лучше показывает себя одиночный инстанс (single)**:

- Пропускная способность выше примерно на 12,5% (4031 vs 3577 ops/sec).
- Средняя и медианная задержка ниже (1,97 и 1,79 мс против 2,21 и 2,08 мс).
- Но у sharded разброс задержек по выборке больше (часть запросов заметно дольше из-за маршрутизации и конфигурации кластера).

Для текущей учебной задачи (небольшая БД, один хост) одиночный инстанс даёт более простую эксплуатацию и лучшую производительность. Шардированный кластер в такой конфигурации проигрывает из-за накладных расходов (mongos, обмен с config server, маршрутизация), при этом оба шарда и конфиг работают на том же железе, поэтому реального выигрыша по ресурсам нет.

**Когда уместен шардинг:** при росте объёма данных и/или при переносе шардов на отдельные серверы. Тогда нагрузка и объём распределяются по узлам, а одиночный инстанс станет узким местом по памяти и CPU. Для запросов по `student_id` (targeted) шардинг по hashed `student_id` сохраняет предсказуемую задержку и масштабируемость.

**Как можно улучшить Шардированный кластер:** разнести шарды (и при необходимости config server) по разным физическим или виртуальным машинам, чтобы снять конкуренцию за CPU/диск и получить реальное горизонтальное масштабирование.
